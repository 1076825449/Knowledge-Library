## 项目一句话总定义

本项目最终要建设的，不是一个税务文件展示站，也不是简单问答集合，而是一个 **基于严谨政策依据、面向真实问题、覆盖企业全生命周期、用大白话解释规则、并能持续扩展到网站检索与 AI 问答的结构化税务知识库系统**。

---

## 1. 项目背景与建设动机

税务知识的难点，往往不在理论深度，而在于：

- 规则数量多
- 口径分散
- 场景复杂
- 文件分布零散
- 同一个问题往往涉及多个政策依据
- 很多实务问题不是简单查一个文件就能解决

现实中，无论是纳税人、财务人员，还是税务系统内部工作人员，常常遇到以下困难：

- 知道自己遇到了问题，但不知道该查什么
- 知道一些文件名字，但不知道哪一条才真正相关
- 能找到政策原文，但看不懂、用不好
- 看到一个答案，却不知道适用边界和例外情形
- 新问题不断出现，旧知识库结构又不便于持续扩展

**因此，本项目不做"政策堆砌型资料库"，而要建设一个：以真实问题为入口，以大白话结论为第一表达层，以政策依据为支撑层，以企业全生命周期为组织主线的结构化税务知识库网站。**

## 2. 项目愿景与最终目标

### 2.1 愿景

建设一个覆盖企业从设立到注销全过程的税务问题知识网站，使用户在面对具体问题时，能够：

1. 快速找到相关问题
2. 先看懂结论
3. 再理解逻辑
4. 最后核验依据
5. 同时看到边界、风险和关联问题

### 2.2 最终目标

本项目最终应发展为一个完整的知识产品，具备以下能力：

| 能力 | 说明 |
|------|------|
| **A. 全生命周期问题覆盖** | 设立期 → 开业期 → 日常经营期 → 变更期 → 风险异常期 → 停业期 → 注销期 |
| **B. 问题卡片式知识组织** | 每个知识项都是一张独立问题卡片，而不是长篇文章或政策摘录 |
| **C. 双层表达机制** | 第一层：大白话答案；第二层：政策依据与边界说明 |
| **D. 网站化使用体验** | 搜索、筛选、标签浏览、导航、关联跳转、更新记录、政策依据展开 |
| **E. 为 AI 问答和智能检索提供底座** | 相似问题推荐、语义检索、向量索引、基于底库引用回答 |

## 3. 核心设计原则

### 3.1 问题导向，而非文件导向

本项目的组织核心不是"文件名称""文号""税种大类"，而是**用户真实会遇到的问题**。

| 正确入口 | 非核心入口 |
|---------|-----------|
| 企业刚设立，还没经营，要不要申报？ | 某某公告汇编 |
| 没收入是不是就不用报税？ | 增值税法规大全 |
| 企业变更地址后税务上还要做什么？ | 发票管理制度文件汇总 |
| 退款后是否就不用开发票？ | |

文件和政策依据仍然重要，但它们应该是**支撑层，不是首层入口**。

### 3.2 先让人看懂，再让专业人员核验

每个答案必须优先解决"读者能否迅速理解"的问题。但"能看懂"不能以牺牲准确性为代价。

**三层表达结构：**

```
第一层：一句话结论 / 大白话结论（直接回答能不能/要不要/怎么办/算不算）
    ↓
第二层：详细解释（解释为什么是这个结论，关键判断点是什么）
    ↓
第三层：政策依据与边界（依据文件、条款位置、适用条件、例外情形、风险提示）
```

### 3.3 内容与依据分离

问题内容和政策依据**不能混成一块大文本**。必须保证：

- 问题是问题
- 依据是依据
- 二者通过关联关系连接

**好处：** 同一政策可支撑多个问题；同一问题可挂多个依据；后续更新政策时不需要重写所有答案；更利于网站和 AI 调用。

### 3.4 稳定骨架，灵活增长

```
稳定骨架：生命周期阶段 × 主题模块 × 问题卡片
灵活增长：标签 × 关联问题 × 更新记录 × 地方口径
```

即：**上层少改，新内容主要通过新增问题和关系扩展。**

### 3.5 优先结构化，避免"写成散文"

本项目最终是一个知识系统，不是若干篇文章。必须做到：字段化、结构化、可查询、可筛选、可扩展、可复用。

### 3.6 不追求一开始"全覆盖"，先追求"正确结构 + 最小可用"

当前阶段重点不是一口气写完所有问题，而是：

1. 把结构搭对
2. 把模板定好
3. 把前几十条高频问题做扎实
4. 让后续新增问题能自然进入系统

## 7. 内容模型设计

本项目最核心的内容单元是**问题卡片（Question Card）**。

一个问题卡片必须是：
- 可单独成立的问题
- 有相对稳定的回答逻辑
- 可以被检索、被筛选、被关联
- 适合长期维护

**它不是：大而空的专题、无边界的经验总结、无依据的问答口径**

## 9. 大白话答案写作规范

### 9.1 基本要求

一句话结论必须做到：先回答问题、不绕圈子、不堆术语、不装专业、但也不能失真。

### 9.2 推荐表达方式

| 类型 | 示例 |
|------|------|
| **直接肯定型** | 企业变更地址后，通常仍需同步办理税务信息变更。 |
| **直接否定型** | 没有收入，不等于一定不用申报。 |
| **条件判断型** | 退款后是否还需要开发票，关键看交易是否成立、款项是否退回以及原发票是否已开具。 |
| **风险提醒型** | 长期零申报不必然违法，但如果与实际经营明显不符，可能引发风险核查。 |

### 9.3 避免的写法

- 绝对化表述
- 空泛口号式表述
- 政策原文照抄
- 仅有结论，没有判断关键
- 模棱两可到无法执行

### 9.4 推荐结构（三段式）

```
第一段：一句话结论
第二段：详细解释
第三段：依据和边界说明
```

## 10. 政策依据管理规范

### 10.1 依据必须单独管理

每条依据单独存储，至少包含：

- 文件名称
- 文号
- 条款位置
- 文件层级
- 生效状态
- 摘要说明

### 10.2 依据和问题之间是多对多关系

- 一个问题可能需要多个依据共同支撑
- 一个依据也可能被多个问题复用

### 10.3 依据引用不能只列文号

必须说明：
- 它支撑的是什么结论
- 是直接依据还是辅助依据
- 是定义依据、程序依据还是风险判断依据

## 11. 标签、关联问题与扩展机制

### 11.1 标签设计目的

标签用来解决两个问题：交叉归类、后续扩展不改骨架。

**示例：** 一个问题可同时带：零申报、小规模纳税人、高频问题、新手必看、广西、发票管理

### 11.2 关联问题设计目的

关联问题用于形成路径式阅读和问题树。

**关系类型：** 下一步问题、相似问题、易混淆问题、延伸问题、进阶问题

### 11.3 新问题进入系统的三种方式

| 方式 | 说明 |
|------|------|
| **A. 新独立问题** | 直接建新卡片 |
| **B. 原问题的子问题** | 根据情况挂为关联问题或子问题 |
| **C. 原问题边界补充** | 更新原卡片而不是新建问题 |

## 16. 地方口径与争议问题管理

### 16.1 地方口径不能污染全国性主答案

处理方式：全国性规则写主答案，地方差异通过 `local_rule_note` 展示。

### 16.2 争议问题要显式标注

**结论稳定度（answer_certainty）：**
- `certain_clear` - 明确
- `certain_condition` - 条件判断
- `certain_dispute` - 存在争议
- `certain_practice` - 偏实务口径

**不要把不稳定结论伪装成绝对规则。**

## 18. 技术实现建议

### 18.1 当前最适合的底座

- **数据库：** SQLite 作为初始数据库，后期可迁移到 MySQL/PostgreSQL
- **前端：** 先做轻量原型

### 18.2 为什么先用 SQLite

- 起步轻
- 容易本地开发
- 适合原型阶段
- 便于 agent 快速建表和调试

### 18.3 前端原则

- 先轻量、可用、结构清晰
- 不要一上来做复杂动画
- 不要把主要精力浪费在视觉炫技上
- 优先把搜索、列表、详情页、筛选打通

### 18.4 后端原则

- 先提供稳定的数据读取
- 明确 API 或数据访问层
- 避免前端直接耦合数据库结构细节

## 24. 附录B：开发原则

1. **优先正确的数据模型，而不是先做漂亮页面**
2. **优先可维护，而不是过早复杂化**
3. **优先结构化，而不是文章化**
4. **优先可追溯，而不是只追求"像是对的"**
5. **优先最小可用，再逐步扩展**
