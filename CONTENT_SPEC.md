# CONTENT_SPEC.md

## 1. 文件目的

本文件用于规定"企业全生命周期税务问题知识库"中每一条知识项的内容写法、字段含义、表达标准和质量要求。

本文件的目标是确保：

- 不同人录入内容时风格尽量统一
- 问题卡片写法结构稳定
- 大白话答案既易懂又不失真
- 每条内容都能挂接政策依据
- 后续网站展示和 AI 检索有统一内容底座

本文件适用于：
- 人工撰写知识内容
- agent 批量生成内容草稿
- 后续编辑、审核、修订知识条目

---

## 2. 基本原则

### 2.1 以问题为单位，不以文章为单位

知识库的最小内容单元是**问题卡片**。

每条内容必须围绕一个明确的问题展开，而不是围绕一个大而空的主题展开。

| 正确示例 | 不推荐示例 |
|---------|-----------|
| 企业没有收入，是否必须申报？ | 发票管理相关问题汇总 |
| 退款后是否就不用开发票？ | 增值税基础知识 |
| 企业变更地址后税务上还要做什么？ | 税务登记制度解读 |

### 2.2 先回答问题，再解释原因

每条知识项必须优先解决用户最关心的事：

- 能不能
- 要不要
- 怎么办
- 算不算
- 有什么风险

**不要**一上来铺政策背景，不要先绕定义，不要先讲制度沿革。

### 2.3 易懂不等于随意

所谓"大白话"，不是降低准确性，也不是随口化表达，而是：

> 用最容易理解的话，准确表达有依据、有限定条件、可执行的结论。

### 2.4 明确边界比写得绝对更重要

税务问题很多都存在：
- 适用条件
- 特殊情形
- 地方差异
- 征管口径差异
- 争议空间

**宁可清楚写明"关键看什么"，也不要写成绝对规则。**

## 3. 一条标准问题卡片应包含什么

| 序号 | 字段 | 说明 |
|------|------|------|
| 1 | 问题编号 | 唯一编码，如 SET-REG-001 |
| 2 | 生命周期阶段 | 设立期/开业期/变更期等 |
| 3 | 主题模块 | 登记管理/申报纳税/发票管理等 |
| 4 | 问题标题 | 简短、像人提问 |
| 5 | 问题完整表述 | 补充问题背景 |
| 6 | 一句话结论 | **最重要字段之一** |
| 7 | 详细解答 | 解释逻辑和判断重点 |
| 8 | 核心定义 | 解释相关术语 |
| 9 | 适用条件 | 答案成立的前提 |
| 10 | 例外与边界 | 重要，不能省略 |
| 11 | 实务处理步骤 | 面向执行 |
| 12 | 风险提示 | 避免的后果 |
| 13 | 政策依据 | 支撑结论，可追溯 |
| 14 | 标签 | 归类和检索增强 |
| 15 | 适用地区 | 全国/地方 |
| 16 | 结论稳定度 | 明确/条件判断/争议/实务口径 |
| 17 | 当前状态 | 草稿/待复核/启用/过时/归档 |
| 18 | 版本号 | 追踪变更 |
| 19 | 更新时间 | 记录时效 |

## 4. 各字段写作规范

### 4.1 question_title（问题标题）

**作用：** 列表展示、搜索命中、详情页标题、用户第一眼理解问题

**要求：**
- 尽量用真实提问语言
- 简洁明确
- 避免公文式标题
- 避免过大、过泛
- 尽量让人一看就知道在问什么

| 好的例子 | 不好的例子 |
|---------|-----------|
| 企业没有收入，是否必须申报？ | 企业涉税问题研究 |
| 企业变更地址后税务上还要做什么？ | 关于税务申报有关事项 |
| 退款后是否就不用开发票？ | 发票处理实务探讨 |
| 员工垫付款，发票能不能开公司抬头？ | 增值税概论 |

---

### 4.2 question_plain（问题完整表述）

**作用：** 补充问题背景，便于准确理解问题边界

**要求：**
- 比标题更完整
- 说明关键事实
- 不要写成长篇案例
- 重点写"这个问题是在什么情境下问的"

**示例：**
> 企业已经办好营业执照，但暂时没有开展经营活动，也没有收入。在这种情况下，是否仍需要按规定办理申报？

---

### 4.3 one_line_answer（一句话结论）

**作用：** 知识项的第一表达层，是最重要的字段之一

**要求：**
- 一眼能看懂
- 直接回答核心问题
- 尽量控制在 1～3 句话
- 能先给结论，再提示判断关键
- 不能失真
- 不轻易绝对化

**推荐模板：**

| 类型 | 示例 |
|------|------|
| **直接肯定型** | 企业变更地址后，通常仍需同步办理税务信息变更。 |
| **直接否定型** | 没有收入，不等于一定不用申报。 |
| **条件判断型** | 退款后是否还需要开发票，关键看交易是否成立、款项是否退回以及原发票是否已开具。 |
| **风险提醒型** | 长期零申报不必然违法，但如果与实际经营明显不符，可能引发风险核查。 |

**禁止写法：**
- 纯政策原文复制
- 只有口号，没有结论
- 过度绝对化表述
- 写了很多，但没有明确回答问题

---

### 4.4 detailed_answer（详细解答）

**作用：** 解释为什么是一句话结论，补足逻辑和实际理解路径

**要求：**
- 解释关键逻辑
- 说明判断重点
- 避免只重复一句话结论
- 语言仍应可理解，不要堆砌术语
- 可以适度使用专业术语，但首次出现时尽量解释

**推荐结构：**
1. 为什么不能简单这样理解
2. 实际判断时要看什么
3. 不同情况可能有什么不同结果

---

### 4.5 core_definition（核心定义）

**作用：** 解释该问题涉及的专业术语，降低理解门槛

**要求：**
- 只解释和本问题真正相关的术语
- 定义简明清楚
- 先专业术语，再白话解释
- 不要堆很多无关概念

**适合解释的内容示例：**
- 零申报
- 税费种认定
- 非正常户
- 视同销售
- 税前扣除
- 简易计税
- 红字发票

**示例：**
> **零申报**：指纳税人在某一申报期内虽无应税收入、无应纳税额或无计税依据，但依法仍填报零数据完成申报的行为。

---

### 4.6 applicable_conditions（适用条件）

**作用：** 说明当前答案成立的前提

**要求：**
- 写明"在哪些情形下适用"
- 不能省略
- 要尽量具体
- 不要只写"视情况而定"

**示例：**
> 适用于已完成设立登记并进入税务管理状态，且相关税费项目已认定或已产生申报义务的经营主体。

---

### 4.7 exceptions_boundary（例外与边界）

**作用：** 防止把有条件规则理解成绝对规则

**要求：**
- 明确哪些情况不适用
- 明确哪些情况需要另行判断
- 明确可能存在的例外
- 对争议点、地方差异、特殊状态进行说明

**示例：**
> 是否必须申报，还需结合停业、注销、非正常户、未认定税费种等特殊状态判断，不能一概而论。

**重要性：** 这一栏是知识库最有价值的部分之一，**不能省略、不能敷衍**。

---

### 4.8 practical_steps（实务处理步骤）

**作用：** 告诉使用者遇到这个问题时，现实中可以怎么操作

**要求：**
- 写成步骤式
- 面向执行
- 不要只讲原则
- 步骤数一般 3～6 步较合适

**示例：**
1. 先确认是否已认定税费种
2. 再确认当前是否已经进入申报期
3. 登录电子税务局查看是否存在待申报事项
4. 如涉及异常状态，先处理异常后再补申报

---

### 4.9 risk_warning（风险提示）

**作用：** 帮助使用者知道"如果处理不当，可能有什么后果或风险"

**要求：**
- 提醒实际风险
- 不故意夸大
- 不泛泛而谈
- 尽量写成业务上可理解的风险

**示例：**
> 长期零申报但实际存在经营活动，可能触发风险预警或后续核查。

---

### 4.10 policy_basis（政策依据）

**作用：** 支撑结论，保证可追溯、可复核

**要求：**
- 不能只写文件名称
- 尽量写到条款位置
- 最好说明该依据支撑的是哪部分结论
- 避免只列一串文号而不解释用途

**推荐写法：**
- 文件名称
- 文号
- 条款位置
- 支撑内容说明

**示例：**
- 《中华人民共和国税收征收管理法》第二十五条：用于支撑纳税人应按规定办理申报义务的基本结论。
- 某公告第X条：用于支撑特定业务情形下的具体处理规则。

---

### 4.11 tags（标签）

**作用：** 交叉归类、搜索增强、关联推荐、后续扩展

**要求：**
- 标签应反映核心特征
- 不要滥打标签
- 控制在少量高质量标签
- 以可检索和可复用为优先

**标签示例：**
- 零申报
- 小规模纳税人
- 一般纳税人
- 地址变更
- 发票管理
- 高频问题
- 新手必看
- 广西

---

### 4.12 answer_certainty（结论稳定度）

**作用：** 防止将不稳定结论当成明确规则

| 值 | 含义 |
|----|------|
| `certain_clear` | 规则明确 |
| `certain_conditional` | 需结合条件判断 |
| `certain_dispute` | 存在争议 |
| `certain_practice` | 偏征管实务口径 |

**当前数据库实际值**：`certain_clear` / `certain_conditional`（后两项暂未入库）

**要求：** 每条问题尽量标注。

---

### 4.13 scope_level / local_region（适用范围 / 地区）

**作用：** 区分全国通用规则与地方执行差异

**scope_level 值（DB实际值）：**
- `scope_national` - 全国
- `scope_local` - 地方

**scope_level 扩展值（保留用于未来）：**
- `scope_mixed` - 混合

**local_region 示例：**
- 全国
- 广西
- 柳州
- 柳江区

**原则：** 全国性规则和地方性补充要分开，避免混写。

## 6. 问题颗粒度标准

颗粒度太粗，会无法回答；颗粒度太细，会导致知识库爆炸。

### 合适颗粒度的标准
一个问题应尽量满足：
1. 用户真实会问
2. 可以相对稳定地回答
3. 能直接指导处理
4. 不依赖极复杂、极个别的事实组合

| 合适示例 | 不合适示例 |
|---------|-----------|
| 企业没有收入，是否必须申报？ | 增值税怎么处理？ |
| 企业变更地址后税务上还要做什么？ | 税务问题大全 |
| 退款后是否就不用开发票？ | 某企业在某年某月某具体业务中的极细枝末节定性 |
| 长期零申报会不会有风险？ | |

## 7. 问题来源建议

知识项最好优先来自真实高频问题，而不是闭门造题。

**优先来源：**
- 办税服务中的高频咨询
- 纳税人常见误区
- 内部工作人员反复讨论的问题
- 风险应对中的高频争议点
- 系统办理中常见报错和难点
- 变更、注销等环节中的高痛点事项

---

## 8. 大白话表达规则

### 8.1 应优先使用的表达

- 一般情况下
- 通常
- 不当然
- 关键看……
- 不能一概而论
- 需要结合……判断
- 在……情形下

### 8.2 应谨慎使用的表达

除非确有明确法定规则且无明显例外，否则避免：

- 必须
- 一定
- 绝对
- 一律
- 全部
- 肯定

---

## 9. 内容分层规则

建议始终保持三层信息结构：

| 层级 | 内容 | 适合人群 |
|------|------|---------|
| **第一层：快速答案层** | 一句话结论 | 绝大多数人先看 |
| **第二层：解释层** | 说明为什么、关键判断点 | 需要理解逻辑的人 |
| **第三层：支撑层** | 政策依据、边界、风险、地方差异 | 需要专业复核的人 |

---

## 10. 常见写作错误

以下问题应尽量避免：

| 错误 | 问题 | 后果 |
|------|------|------|
| **错误1** | 只堆政策原文 | 没人看得懂 |
| **错误2** | 只有大白话，没有依据 | 缺乏可信度和可复核性 |
| **错误3** | 一句话过于绝对 | 误导使用者 |
| **错误4** | 没有边界和例外 | 看起来像对，但实际容易出错 |
| **错误5** | 标题不像人话 | 搜索体验和阅读体验都差 |
| **错误6** | 把多个问题揉成一个问题 | 结构混乱，不利于检索和关联 |

## 11. 审核标准

一条内容进入正式库前，建议至少检查以下问题：

- [ ] 标题是否像真实用户会问的问题？
- [ ] 一句话结论是否直接回答问题？
- [ ] 详细解答是否补足了逻辑？
- [ ] 是否解释了必要术语？
- [ ] 是否明确了适用条件？
- [ ] 是否明确了例外与边界？
- [ ] 是否给了实务处理路径？
- [ ] 是否提示了风险？
- [ ] 是否有政策依据支撑？
- [ ] 是否写清楚依据支撑什么结论？
- [ ] 是否标注了地区和稳定度？
- [ ] 是否有合适标签？
- [ ] 是否具备更新时间和版本信息？

## 12. 示例模板

### 问题标题
企业没有收入，是否必须申报？

### 问题完整表述
企业已经完成设立登记，但暂时没有开展经营活动，也没有收入。在这种情况下，是否仍需要按规定办理申报？

### 一句话结论
企业没有收入，不当然等于不用申报；关键看是否已发生税费种认定并进入相应申报期。

### 详细解答
企业是否没有收入，与是否没有申报义务不是同一个问题。对于已经认定税费种并进入申报周期的纳税人，即使当期没有收入、没有应纳税额，也可能仍需按规定办理零申报。若相关义务尚未发生，则不能机械理解为一律都要申报。

### 核心定义
**零申报**：是指纳税人在某一申报期内虽无应税收入、无应纳税额或无计税依据，但依法仍填报零数据完成申报的行为。

### 适用条件
适用于已完成设立登记并进入税务管理状态，且相关税费项目已认定或已产生申报义务的经营主体。

### 例外与边界
是否必须申报，还需结合停业、注销、非正常户、未认定税费种等特殊状态判断，不能一概而论。

### 实务处理步骤
1. 先查看是否已认定税费种
2. 再查看当前是否已经进入申报期
3. 登录电子税务局查看是否存在待申报事项
4. 如存在异常状态，先处理异常后再补办申报

### 风险提示
长期零申报但实际存在经营活动，可能引发风险预警或后续核查。

### 政策依据
列明相关法律、公告、条款位置，并说明支撑的具体结论。

## 13. 内容维护原则

### 13.1 新增问题时
优先判断它属于：
- 新独立问题 → 直接新建
- 原问题子问题 → 挂为关联问题
- 原问题边界补充 → 更新原卡片

### 13.2 政策变化时
优先更新：
- 政策依据状态
- 详细解答
- 边界说明
- 更新记录

### 13.3 地方差异出现时
优先通过地方补充结构（`local_rule_note`）增加，**不要轻易改动全国性主答案**。

## 14. 最终目标提醒

本文件服务的最终目标不是"写几篇税务答疑"，而是建设一个：

> **基于问题卡片、能让人看懂、能经得起政策复核、能持续增长并适合网站检索与 AI 扩展的结构化税务知识内容体系。**
