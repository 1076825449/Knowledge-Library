# ROADMAP.md

> **文件性质**：项目执行路线图 + 阶段验收清单
> **适用对象**：所有参与本项目的 agent、自动化编码助手或开发协作者
> **阅读顺序**：`PROJECT_MANIFEST.md` → `ROADMAP.md` → `AGENTS.md` → 具体任务文档
> **目的**：把项目从“已有原型”稳步推进到“最终知识产品”，并防止后续开发跑偏

---

## 1. 本文件解决什么问题

当前项目已经有：

- 数据库底座
- 基础网站原型
- 批量导入脚本
- 质量巡检脚本
- AI 导出预留

但当前项目也存在一个明确风险：

- 文档口径、数据库口径、页面口径、录入口径开始出现分叉

因此，本文件的作用不是重复介绍项目愿景，而是回答下面三个问题：

1. 现在项目真实处于什么阶段
2. 下一步应该先做什么、后做什么
3. 每做完一阶段，怎样判断“真的完成了”

---

## 2. 后续 agent 的强制执行规则

后续所有 agent 在动手前，必须遵守以下顺序：

1. 先读 `PROJECT_MANIFEST.md`
2. 再读 `ROADMAP.md`
3. 再读 `AGENTS.md`
4. 再看当前任务相关文档，如 `TASKS.md`、`CONTENT_SPEC.md`、`README.md`
5. 再检查当前代码、数据库、测试和数据报告
6. 最后才允许做修改

如果 `ROADMAP.md` 与旧的阶段完成标记、旧的 README 描述不一致：

- 以 `PROJECT_MANIFEST.md` 的项目定位为最高准则
- 以 `ROADMAP.md` 的执行顺序和验收口径为当前工作准则
- 以数据库与质量报告为当前事实源

## 3. 当前项目真实基线

以下内容用于帮助后续 agent 正确认知项目状态，避免误判为“从零开始”或“已经全部完成”。

### 3.1 已有基础

- 已存在 8 张核心表及索引
- 已存在 Flask 后端、Jinja 模板前端
- 已存在问题列表页、详情页、首页、新建页、编辑页
- 已存在批量导入和质量巡检脚本
- 已存在 AI 导出脚本
- 已有较大规模问题数据和政策关联数据

### 3.2 当前阶段的真实重点

当前已经不是“先把数据库和原型搭起来”的阶段。

当前最重要的是：

- 统一规范口径
- 修正代码和数据之间的不一致
- 补齐内容结构质量
- 让录入和维护闭环稳定
- 为后续检索、扩容、AI 引用打下稳定底座

### 3.3 当前最典型的风险

- 文档静态数字和数据库真实数字不一致
- 枚举体系在数据库、模板、表单、脚本中不一致
- 搜索在页面、API、脚本中口径不一致
- 内容数量增长快于内容结构质量
- 录入功能已存在，但还不够稳定地服务长期维护

---

## 4. 项目推进的总原则

后续开发必须始终遵守以下顺序：

```text
统一事实源
→ 统一枚举与规则
→ 修复代码口径分叉
→ 补结构质量
→ 补薄弱覆盖
→ 强化录入与维护闭环
→ 完善检索与 AI 导出
→ 最后做产品化和部署闭环
```

### 4.1 永远不要反过来做

不要这样推进：

- 先做更复杂前端
- 先做 AI 问答
- 先做推荐系统
- 先做花哨后台
- 先做大量新增问题

如果基础口径还没统一、结构质量还没补齐，上述工作都容易放大混乱。

---

## 5. 事实源与优先级规则

后续所有 agent 必须按以下优先级判断事实：

1. `PROJECT_MANIFEST.md`：定义项目定位、边界、最终目标
2. `ROADMAP.md`：定义当前阶段执行顺序、验收标准、不得偏离的主线
3. 数据库当前状态：定义真实内容规模、真实字典值、真实结构缺口
4. 质量巡检报告：定义当前内容质量短板
5. `AGENTS.md`：定义行为约束、禁止事项、决策规则
6. `README.md`、`TASKS.md`、`TASKS_PROGRESS.md`：作为说明文档，不得凌驾于数据库事实之上

如果发现 README、任务状态、交接摘要与数据库不一致：

- 不要先相信文档
- 先核对数据库
- 再更新文档

## 6. 最终目标拆解为 8 个执行阶段

---

## 7. 阶段 A：统一项目口径

### 7.1 目标

把“文档、数据库、后端、模板、表单、脚本”的核心口径统一，防止项目继续在分叉状态下扩展。

### 7.2 必做项

- 统一项目事实源说明
- 清理 README 中过期的静态数字和过度乐观的完成状态
- 确认并固定核心枚举体系：
  - `question_type`
  - `answer_certainty`
  - `status`
  - `scope_level`
  - `support_type`
  - `relation_type`
  - `policy_level`
  - `policy_status`
  - `update_type`
- 建立“唯一枚举口径”的来源，不允许模板和表单硬写一套、数据库再跑另一套
- 统一页面查询和 API 查询的参数口径
- 统一页面搜索和 API 搜索的行为逻辑

### 7.3 阶段 A 验收清单

- [ ] README 不再保留明显过期的静态统计数字
- [ ] 数据库中的所有核心枚举值，都能在页面和表单正确显示
- [ ] 新建问题时允许选择的枚举值，与数据库允许值一致
- [ ] 详情页能正确渲染所有合法问题类型、支撑类型、状态、关系类型
- [ ] `/questions` 与 `/api/questions` 使用同一筛选语义
- [ ] `/questions` 与 `/api/search` 不再出现明显搜索行为分叉
- [ ] 后续 agent 可以明确知道“哪个文件是事实源，哪个文件只是说明”

### 7.4 不完成本阶段前，不应优先做的事

- 不要继续大规模扩容新问题
- 不要继续做更复杂前端
- 不要接 AI 问答

---

## 8. 阶段 B：补齐内容结构质量

### 8.1 目标

把现有大量问题从“有标题、有结论”提升为“结构完整、可核验、可维护”的知识卡片。

### 8.2 必做项

- 补齐缺失的 `applicable_conditions`
- 补齐缺失的 `exceptions_boundary`
- 补齐缺失的 `practical_steps`
- 补齐缺失的 `risk_warning`
- 补齐缺失的业务标签
- 补齐缺失的更新记录
- 补齐缺失的关联问题
- 对高频问题建立更严格的结构完整要求

### 8.3 执行优先顺序

先补以下问题，不要平均撒网：

1. 高频问题
2. 新手必看问题
3. 高流量主题链路问题
4. 已有政策支撑但结构字段缺失的问题
5. 缺标签、缺更新、缺关联的问题

### 8.4 阶段 B 验收清单

- [ ] active 问题不再大面积缺 `适用条件`
- [ ] active 问题不再大面积缺 `例外与边界`
- [ ] active 问题不再大面积缺 `实务处理步骤`
- [ ] active 问题不再大面积缺 `风险提示`
- [ ] 所有 active 问题至少有 1 个业务标签
- [ ] 所有 active 问题至少有 1 条更新记录
- [ ] 所有核心高频问题至少有 2 个以上关联问题
- [ ] 质量报告中“结构缺口”从系统性缺口下降为零星缺口

### 8.5 高价值问题的最低标准

对高频或新手问题，至少要满足：

- 有一句话结论
- 有详细解答
- 有核心定义
- 有适用条件
- 有例外与边界
- 有实务处理步骤
- 有风险提示
- 有至少 2 条政策依据
- 有至少 2 个业务标签
- 有至少 2 个关联问题
- 有更新记录

---

## 9. 阶段 C：补齐覆盖短板

### 9.1 目标

不是盲目扩容，而是让“生命周期 × 模块”的骨架不出现明显空洞或严重失衡。

### 9.2 必做项

- 补齐 ETAX 模块
- 检查阶段 × 模块覆盖矩阵
- 找出空白组合或明显过薄组合
- 以真实高频场景扩充，而不是为了凑数扩容

### 9.3 选题优先级

优先补这些方向：

- 电子税务局办理链路
- 零申报与未申报链路
- 地址变更链路
- 红字发票链路
- 非正常户链路
- 注销清税链路
- 小微优惠链路
- 社保费申报链路

### 9.4 阶段 C 验收清单

- [ ] ETAX 模块不再为 0
- [ ] 固定导航中的每个主题模块都具备可用内容
- [ ] 阶段 × 模块矩阵不再出现明显空白入口
- [ ] 每个重点链路至少形成“入口问题 + 流程问题 + 边界问题 + 风险问题”
- [ ] 新增问题不是孤点，而是能挂标签、挂依据、挂关联

---

## 10. 阶段 D：稳定录入与维护闭环

### 10.1 目标

让新增问题更像“填表录入”而不是“改代码”，同时保证录入动作不会破坏结构一致性。

### 10.2 必做项

- 让新建/编辑页面完全对齐标准字段
- 增强政策依据关联能力，不局限于过低数量
- 增加对标签、关联问题、更新记录、地方口径的维护入口
- 支持草稿、待复核、启用的内容状态流转
- 增加录入校验规则和错误提示
- 增加批量导入前校验和导入后审计

### 10.3 阶段 D 验收清单

- [ ] 新建问题无需改代码即可完成标准录入
- [ ] 编辑问题可维护标签、政策依据、关联问题、更新记录
- [ ] 录入时能及时发现必填字段缺失、非法枚举、重复编码等问题
- [ ] 不合格内容不会轻易直接进入 `active`
- [ ] 内容维护者可以按标准字段长期扩充内容

---

## 11. 阶段 E：强化检索与筛选体验

### 11.1 目标

让网站真正成为“查问题工具”，而不是“能搜一下的内容展示页”。

### 11.2 必做项

- 统一搜索逻辑
- 强化同义表达扩展
- 优化标题、关键词、一句话结论的综合命中
- 提升列表页筛选维度和排序稳定性
- 区分 `scope_level` 与 `local_region`
- 提升关联问题与下一步推荐的可用性

### 11.3 第一优先级搜索表达

必须优先保证这些表达能较稳定命中核心问题：

- 没收入要不要报税
- 没经营要不要申报
- 零申报是不是不用报
- 地址变了税务怎么办
- 发票开错了怎么办
- 非正常户怎么解除
- 注销前要先做什么

### 11.4 阶段 E 验收清单

- [ ] 页面搜索和 API 搜索结果基本一致
- [ ] 常见口语化表达能命中核心问题
- [ ] 标签筛选、阶段筛选、模块筛选可同时工作
- [ ] 地区口径筛选不再混淆“适用范围”和“具体地区”
- [ ] 详情页中的关联问题和下一步推荐具备真实帮助

---

## 12. 阶段 F：建立质量守门机制

### 12.1 目标

保证后续再扩容时，项目不会重新掉回“数量增长、结构失控”的状态。

### 12.2 必做项

- 建立统一质量巡检规则
- 把关键质量规则测试化
- 把核心结构约束纳入 CI 或固定脚本
- 为批量导入建立前置校验
- 建立定期审计输出

### 12.3 至少要守住的质量红线

- active 问题不能没有政策依据
- active 问题不能没有业务标签
- active 问题不能没有更新记录
- active 高频问题不能缺条件、边界、步骤、风险
- 非法枚举值不能进入数据库
- 模板和表单不能落后于数据库实际口径

### 12.4 阶段 F 验收清单

- [ ] 已有测试可验证关键字典一致性
- [ ] 已有测试可验证 active 内容的最低质量要求
- [ ] 批量导入脚本不能轻易写入非法枚举或不完整内容
- [ ] 质量报告可持续输出并指导下一批工作

---

## 13. 阶段 G：为 AI 检索和引用式回答打底

### 13.1 目标

先把可导出、可切块、可引用的知识 contract 定稳定，再考虑更高级的智能层。

### 13.2 必做项

- 明确 AI 导出字段 contract
- 明确切块单位是否以问题卡片为主
- 导出时保留 `question_code` 和 `policy_code`
- 保证 AI 返回能引用回原问题和原依据
- 明确哪些字段属于回答层，哪些字段属于支撑层

### 13.3 当前阶段不应做的事

- 不要直接接“AI 自动生成并发布”
- 不要跳过结构治理去上向量库
- 不要把 AI 当成内容底座

### 13.4 阶段 G 验收清单

- [ ] AI 导出结构稳定且可复用
- [ ] 每条导出内容都能追溯回问题卡片
- [ ] 每条导出内容都能追溯回政策依据
- [ ] 不依赖 AI，网站本身也能独立成立

---

## 14. 阶段 H：产品化与部署闭环

### 14.1 目标

确保项目不仅能在本地运行，还能稳定交付、部署、备份和继续维护。

### 14.2 必做项

- 修正默认口令和默认密钥问题
- 明确生产环境变量
- 跑通从空库初始化到网站可访问的全流程
- 校验测试命令在标准环境可执行
- 补齐部署文档、备份文档、恢复文档

### 14.3 阶段 H 验收清单

- [ ] 不再依赖硬编码默认管理密码
- [ ] 不再依赖硬编码默认密钥
- [ ] README 的快速开始可在干净环境复现
- [ ] 测试命令可在标准环境复现
- [ ] 数据库备份与恢复路径明确
- [ ] 生产部署最小流程清晰

## 15. 后续 agent 的任务排序规则

当同一时间有多个需求时，按以下顺序判断先做什么：

1. 是否会破坏项目定位
2. 是否会破坏数据结构一致性
3. 是否能消除口径分叉
4. 是否能补高价值结构缺口
5. 是否能提升长期维护性
6. 是否能补重点链路覆盖
7. 是否能提升检索体验
8. 是否只是视觉或次要增强

如果一个需求更炫、更显眼，但不能增强结构质量或维护能力，应延后。

## 16. 后续 agent 的禁止事项

没有明确授权时，禁止做以下事情：

- 把知识库改造成文章博客系统
- 把问题内容和政策依据重新写回一个大字段
- 新增大量页面逻辑却不修正底层枚举和查询口径
- 为了好看修改骨架分类
- 为了快速扩容牺牲字段完整性
- 用 AI 自动生成内容直接入库
- 把 README 的阶段状态改得比真实进度更乐观
- 仅凭旧交接文档判断项目是否完成

## 17. 每次任务结束后的最小交付要求

每个 agent 完成阶段性工作后，至少应交付下面几类内容中的相关部分：

### 17.1 如果改了数据层

- 建表/迁移 SQL
- 字典或初始化数据变更
- 结构变更原因
- 对现有数据的兼容说明

### 17.2 如果改了页面层

- 页面行为说明
- 所依赖的数据读取逻辑
- 是否影响录入或检索路径

### 17.3 如果改了内容层

- 补了哪些问题
- 补了哪些结构字段
- 是否补了标签、政策、关联、更新记录
- 质量缺口是否下降

### 17.4 如果改了规则层

- 更新哪些文档
- 统一了哪些枚举或校验口径
- 后续 agent 需要注意什么

## 18. 最终完成标准

项目接近最终目标时，至少应满足以下判断：

### 18.1 结构层

- [ ] 问题、政策、标签、更新、关联、地方口径结构稳定
- [ ] 枚举体系统一，页面/API/录入/脚本一致
- [ ] 录入和导出都围绕问题卡片展开

### 18.2 内容层

- [ ] 核心高频问题链路已完整覆盖
- [ ] active 内容具备基本结构完整性
- [ ] 主要问题都可追溯到政策依据
- [ ] 争议或地方差异有显式标注

### 18.3 网站层

- [ ] 首页是知识导航首页，不是营销首页
- [ ] 列表页是问题卡片列表，不是文章列表
- [ ] 详情页是整个产品的核心页面
- [ ] 搜索、筛选、标签、关联跳转都稳定可用

### 18.4 维护层

- [ ] 新增内容更接近表单录入而不是改代码
- [ ] 有固定质量巡检机制
- [ ] 有测试和文档守住核心规则

### 18.5 AI 层

- [ ] AI 只是增强层，不是底座
- [ ] 导出数据可切块、可追溯、可引用
- [ ] 不依赖 AI，知识库网站本身仍然成立

---

## 19. 一句话执行纲领

**先统一口径，再补结构质量，再补覆盖短板，再强化录入和检索，最后做 AI 增强与产品化闭环。**
